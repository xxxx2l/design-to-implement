# design-to-implement

> 一个 agent 技能（skill）：把"方案讨论"到"代码落地"的完整研发流程沉淀成结构化文档工作流。

[![Skill Type](https://img.shields.io/badge/type-agent%20skill-blue)](#) [![License](https://img.shields.io/badge/license-MIT-green)](#license)

## 这是什么

一个为 AI coding agent 设计的技能（skill），强制把"想"和"做"分开：

- **讨论阶段**：只对话，不写代码，结论沉淀到沟通过程文档
- **设计阶段**：把共识整理成设计文档（含验收标准），作为唯一事实来源
- **实施规划阶段**：根据设计拆出实施步骤 + 测试用例
- **实施阶段**：按步骤落地，每步对照测试用例验证

核心理念是**先讨论清楚，再动手写代码**。讨论没有结论、用户没有明确说"开始落地"，绝不动代码。

## 解决什么问题

在 AI agent 协助开发时，常见痛点：

- 还没想清楚就开始写代码，写到一半发现方向错了
- 跨多次对话/上下文时，之前的讨论结论丢了，反复重读代码或要求用户复述
- 方案变更时，设计和实现脱节，文档变成事后记录
- 实施完才发现没明确的"做完了"标准，测试靠感觉
- 大任务在一个上下文里跑完，上下文爆炸，后期质量下降

本技能用一套结构化文档工作流 + 多上下文记忆机制系统性地解决这些问题。

## 核心机制

| 机制 | 说明 |
|------|------|
| 四阶段工作流 | 讨论 → 设计 → 实施规划 → 实施，每阶段切换需用户显式确认 |
| 多上下文记忆 | 每个任务有"任务状态"文档作为记忆锚点，新对话进入先读它恢复上下文 |
| 单一事实来源 | 方案变更永远改设计文档（含变更记录），不直接改实施文档 |
| 验收标准 + 测试用例 | 设计阶段定验收标准（规则），实施规划阶段落成测试用例（实例），三层职责分离 |
| 修改 / 回退机制 | 实施中发现方案缺口走"修改"（轻量增量），发现根本错误走"回退"（推倒重来），都需用户确认 |
| 任务拆分（模式 A） | 大型多模块任务可拆成父任务（接口契约）+ 子任务（各自完整流程） |
| 双 agent 实施（模式 B） | 单任务实施+测试阶段可拆给两个独立 agent 串行执行，上下文隔离 |

## 适用场景

- 中大型功能的方案设计 + 实施
- 需要跨多次对话推进的研发任务
- 后端项目（设计文档强制包含"接口变动清单"供前端联调）
- 多模块的复杂系统设计

**不适合**：一次性小需求、改个文案、加个字段——这种直接做就行，别套流程。

## 工具兼容性

**技能本体工具中立**——核心工作流（文档维护、阶段切换、变更机制）不依赖任何具体 agent 客户端，只要客户端能"读本地文件 + 写本地文件"就能用。

扩展模式（A/B）涉及多 agent 调度，调度实现与客户端相关：

| 客户端 | 支持情况 |
|--------|---------|
| ZCode | 已验证，具体做法见 [`assets/references/zcode-scheduling.md`](assets/references/zcode-scheduling.md) |
| 其它客户端 | 未验证，欢迎贡献适配参考（见下方"贡献"） |

## 安装

把本目录放到 agent 客户端识别的 skills 目录下，常见路径：

- ZCode：`~/.agents/skills/design-to-implement/` 或 `<project>/.agents/skills/design-to-implement/`
- 其它遵循 `.agents/skills/` 约定的客户端：同上

放好后，在对话中显式调用即可（如 `/design-to-implement` 或 "用 design-to-implement 技能来处理这个需求"）。本技能为**纯手动触发**，不会根据关键词自动激活。

## 使用示例

启动一个新任务（在 agent 对话中）：

```
用户：/design-to-implement
      我们要做一个文件服务 SDK 的封装，把现有 HTTP 接口转成 SDK

agent：（创建 docs/design/01-文件服务SDK封装/ 文件夹，生成 5 份文档骨架，
        进入讨论阶段，列出未决疑问清单）
        我需要先和你确认几点：1. SDK 语言？2. 是否保留原有 HTTP 接口？...

用户：（逐项回答）

agent：（更新沟通过程文档，标记已闭环的疑问）
        所有疑问已闭环，是否进入设计阶段？

用户：开始设计

agent：（生成 02-设计.md，含模块划分 + 接口契约 + 验收标准）
        ...
```

完整流程见 SKILL.md。

## 结构

### 技能包结构（clone 后你会看到什么）

```
design-to-implement/
├── README.md                          ← 项目说明（你正在读）
├── LICENSE                            ← MIT
├── SKILL.md                           ← 技能主体（agent 加载后阅读）
└── assets/
    ├── readme.md                      ← 任务总览模板
    ├── task-status.md                 ← 任务状态锚点（普通/子任务）
    ├── parent-task-status.md          ← 父任务专用（A 模式）
    ├── communication.md               ← 沟通过程模板
    ├── design.md                      ← 设计文档模板（含验收标准区块）
    ├── implementation.md              ← 实施文档模板（步骤 + 用例引用）
    ├── test.md                        ← 测试用例模板（四层用例）
    └── references/
        └── zcode-scheduling.md        ← ZCode 调度参考（按需加载）
```

**Progressive disclosure 约定**：
- `SKILL.md` 永远在上下文里（agent 加载技能即读）
- `assets/` 下的模板按需读取（创建对应文档时才读）
- `assets/references/` 下的参考文件进一步按需加载（仅 B 模式 + 特定客户端时才读）

### 用户项目产物结构（使用技能后会得到什么）

```
你的项目/
└── docs/design/
    ├── README.md                          ← 全局任务总览（所有任务的入口）
    │
    ├── 01-单模块任务/                      ← 普通任务（4 阶段文档各一份）
    │   ├── 00-任务状态.md                  ← 记忆锚点（新对话先读这个）
    │   ├── 01-沟通过程.md                  ← 讨论结论 + 未决疑问清单
    │   ├── 02-设计.md                      ← 方案 + 验收标准 + 接口变动清单
    │   ├── 03-实施.md                      ← 步骤清单（每步引用测试用例）
    │   └── 04-测试.md                      ← 四层用例集（F/EX/R/API）
    │
    └── 02-大型多模块任务/                  ← 父任务（A 模式）
        ├── 00-任务状态.md                  ← 含子任务清单 + 依赖关系
        ├── 02-设计.md                      ← 总体架构 + 模块拆分 + 接口契约
        └── 子任务/
            ├── 02-1-子模块A/               ← 子任务（用普通模板，完整 4 阶段）
            │   └── ...（00~04 文档）
            └── 02-2-子模块B/
                └── ...
```

**记忆机制示意**（新对话进入任务时，agent 按此顺序恢复上下文）：

```
1. docs/design/README.md              → 定位当前任务
2. 任务/00-任务状态.md                 → 确认当前阶段和下一步
3. 当前阶段对应的文档（如 02-设计.md） → 接续工作
4. 只在需要时读代码                    → 不作为恢复上下文的第一手段
```
