# ZCode 调度实现参考

> 本文是 B.5「agent 调度指引」在 **ZCode** 客户端中的具体落地参考。SKILL.md 保持工具中立，只定义交接协议和角色职责；本文补充 ZCode 当前能力下推荐的调度做法。
>
> 如果你用的不是 ZCode，可以参照本文思路，对照你所用客户端的子 agent / 后台任务 / 多窗口能力做适配。
>
> **欢迎贡献**：如果你为其它客户端（如 Claude Code、Cursor 等）写了对应的调度适配参考，欢迎在 `assets/references/` 下新增 `<客户端名>-scheduling.md` 并提交 PR。命名约定：全小写、连字符分隔，如 `claude-code-scheduling.md`。

## 能力前提

ZCode 当前（截至本文编写）提供以下与多 agent 相关的能力：

- **子 agent 调用**：主 agent 可通过 Agent 工具启动子 agent，传入路径和指令；子 agent 干完返回结果，其上下文随之销毁
- **后台运行**：`run_in_background: true` 可让子 agent 异步执行，完成时通知主 agent
- **多对话窗口**：用户可开多个独立的 ZCode 会话窗口，各自维护上下文

下面三种做法对应这三种能力。

## 做法 1（推荐）：主 agent 作为调度中心

最贴合 ZCode 能力的 B 模式落地方式。主 agent（你正在对话的那个 agent）持有全局视野，负责读状态文档、判断当前环节、派发任务给子 agent：

```
用户："启用 B 模式"
   ↓
主 agent 读 00-任务状态.md 的"实施分段状态"区块
   ↓
启动子 agent A（实现）—— 用 Agent 工具，传入设计/实施文档路径 +
   指令"按 03-实施.md 推进实现，完成后更新实施分段状态区块，声明可进入测试"
   ↓
子 agent A 干完返回（其上下文随之销毁，实现细节不污染后续）
   ↓
主 agent 读状态区块，确认"实现已完成"
   ↓
启动子 agent B（测试）—— 用 Agent 工具，传入测试文档路径 +
   指令"按 04-测试.md 跑用例，结果（含失败判定）写入实施分段状态区块"
   ↓
子 agent B 跑完返回
   ↓
主 agent 读状态区块，判断：
   - 失败为实现 bug → 再启动子 agent A（实现）修复，修完回到测试 agent 复测
   - 失败为设计 bug → 停下，报告用户走修改机制
   - 全部通过 → 收尾
```

**关键点**：
- **子 agent 是无状态临时工**，每次 Agent 工具调用都重新建立上下文——这正是 B 模式想要的"上下文隔离"，实现细节不会污染测试上下文，反之亦然
- **主 agent 是调度中心**，负责读文档判断状态、决定下一步——它就是 B 模式事实上的"编排框架"
- 子 agent 之间**不直接感知彼此**，只通过"实施分段状态"区块这个共享文档间接交接
- 完全用 ZCode 现有能力（Agent 工具 + 文件系统），不需要任何外部框架

## 做法 2（最透明）：多对话窗口

如果你想最大程度掌控节奏，不用子 agent，而是用多个独立的 ZCode 对话窗口：

- 窗口 1（实现 agent）："我是任务 X 的实现 agent，按 03-实施.md 推进，完成后在 00-任务状态.md 声明可进入测试"
- 你切到窗口 2（测试 agent）："我是任务 X 的测试 agent，先读 00-任务状态.md 看实现是否完成，然后按 04-测试.md 跑用例"
- 有 bug 则切换或开新窗口继续

好处：最透明，你完全掌控。坏处：手动切换负担重。

## 做法 3（A 模式专用）：后台 agent 并行

A 模式（任务拆分）的多个无依赖子任务可用 `run_in_background: true` 真正并行：主 agent 启动多个后台子 agent 各自跑完整 4 阶段，完成时通知主 agent。**但这种方式不适合 B 模式**——B 的实现和测试是严格串行的，用后台并行反而绕弯。

## 边界说明

技能定义的是**交接协议**（实施分段状态区块的格式和语义）和**角色职责**（实现/测试各做什么），不锁定具体怎么调度。上述做法是 ZCode 当前能力下的推荐方式；如果未来 ZCode 出了更原生的多 agent 编排能力，做法可随之调整，技能本身不需要改。
